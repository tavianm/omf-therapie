## Source

Issue #58: "Actuellement, uniquement les mercredis sont éligibles à la prise de rendez-vous au cabinet. Le thérapeute aurait besoin de rajouter manuellement des plages de temps éligibles au rdv au cabinet."

## Problem

Code actuel restreint les RDV cabinet au mercredi uniquement (ligne 342-346 dans `src/lib/google-calendar.ts`):

```typescript
// Présentiel : uniquement le mercredi (ISO 3)
if (mode === 'in-person' && !isWednesday) {
  continue;
}
```

Cela limite la flexibilité : pas de RDV cabinet sur d'autres jours même si disponible.

## Outcome

Thérapeute peut ajouter/modifier/supprimer des plages horaires éligibles pour RDV cabinet. Le système de prise de RDV utilise ces plages + Google Calendar Freebusy pour afficher les créneaux.

## Appetite

**1 semaine** — implémentation rapide, scope minimal.

## Shapes

### Shape 1: Fichier de config (JSON/TS)

**Description:** Ajouter `src/config/manual-slots.ts` avec tableau de jours+periodes.

**Trade-offs:**
- Pro: Simple, pas de DB additionnelle
- Con: Rebuild + deploy nécessaire à chaque changement (commit → CI → Netlify)

**Rough scope:** XS

### Shape 2: Supabase table + UI admin ✓

**Description:** Nouvelle table `manual_time_slots` + composant React dans `/mes-rdvs`.

**Trade-offs:**
- Pro: Dynamique, pas de rebuild, utilise infra existante
- Con: +1 table, UI admin à construire

**Rough scope:** S

### Shape 3: CMS headless

**Description:** Strapi/Contentful pour gestion des slots.

**Trade-offs:**
- Pro: UI admin prête
- Con: Overkill, dépendance externe, coût

**Rough scope:** M

## Fit Check

**Selected: Shape 2 (Supabase table)**

- Site déjà utilise Supabase pour `appointments`
- UI admin existe déjà (`/mes-rdvs`)
- 1 semaine → Shape 1 trop limité (rebuild), Shape 3 trop complexe

**Eliminated:**
- Shape 1: Rebuild requis non-viable pour usage fréquent
- Shape 3: Overkill, ajout de dépendance lourde

## Table Schema (manual_time_slots)

```sql
CREATE TABLE manual_time_slots (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slot_date DATE NOT NULL,           -- YYYY-MM-DD
  period TEXT NOT NULL,              -- 'morning' | 'afternoon' | 'all_day'
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  deleted_at TIMESTAMPTZ NULL        -- soft delete
);

-- Index for date range queries
CREATE INDEX idx_manual_slots_date ON manual_time_slots(slot_date) WHERE deleted_at IS NULL;

-- RLS: seul admin peut CRUD
ALTER TABLE manual_time_slots ENABLE ROW LEVEL SECURITY;
CREATE POLICY admin_full_access ON manual_time_slots
  FOR ALL USING (auth.uid() IN (SELECT id FROM users WHERE role = 'admin'));
```

## Cache Invalidation

CRUD sur `manual_time_slots` → invalidation cache disponibilité:

```typescript
// src/lib/manual-slots.ts
async function invalidateSlotCache() {
  // Supabase: clear cache entries avec prefix 'availability:'
  await supabaseAdmin.from('cache_keys').delete().like('key', 'availability:%');
}
```

Appelé après POST/PATCH/DELETE sur time-slots API.

## Data Flow Complété

1. **Admin UI** → POST `/api/admin/time-slots` → Supabase
2. **Cache invalidation** → clear availability cache
3. **Booking flow** → GET `/api/availability?mode=in-person`
4. **`generateCandidateSlots()`** modifié:
   - Fetch `manual_time_slots` depuis Supabase
   - Merge avec logique mercredi existante (union)
   - Return slots candidats augmentés
