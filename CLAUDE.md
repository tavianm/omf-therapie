## Project Overview

OMF Therapie is a professional therapy practice website built with Astro 5 and TypeScript. The site emphasizes accessibility (WCAG 2.1 AA compliance), performance optimization through Islands Architecture, and educational content through a blog system.

**Tech Stack:** Astro 5 + TypeScript + React Islands + Tailwind CSS

## Architecture & Patterns

### Routing System

The app uses **Astro file-based routing** — each `.astro` file in `src/pages/` maps directly to a URL:

```
src/pages/
├── index.astro              → /
├── a-propos.astro           → /a-propos
├── contact.astro            → /contact
├── accessibilite.astro      → /accessibilite
├── 404.astro                → /404
├── blog/
│   ├── index.astro          → /blog
│   └── [slug].astro         → /blog/:slug  (static pre-rendered)
└── services/
    ├── therapie-individuelle.astro   → /services/therapie-individuelle
    ├── therapie-de-couple.astro      → /services/therapie-de-couple
    ├── therapie-familiale.astro      → /services/therapie-familiale
    └── troubles-alimentaires.astro   → /services/troubles-alimentaires
```

Legacy SPA paths (`/Tarifs`, `/Services`, `/About`, etc.) are handled via `[[redirects]]` in `netlify.toml`.

**Layouts:**
- `src/layouts/Layout.astro` — base layout with SEO head, JSON-LD, nav, footer
- `src/layouts/ServiceLayout.astro` — shared layout for service pages

### Blog System

Blog posts use **Astro Content Collections** (Markdown files, not TypeScript modules):

1. **Post Storage:** `src/content/blog/*.md` — Markdown with YAML frontmatter
2. **Schema:** Defined in `src/content.config.ts` with Zod validation
3. **Rendering:** `src/pages/blog/[slug].astro` — statically pre-rendered at build time using `entry.render()` + `<Content />`
4. **Listing:** `src/components/blog/BlogClientWrapper.tsx` — React island for client-side search/filter

**Adding a new blog post:**
1. Create `src/content/blog/your-slug.md` with required frontmatter
2. No registry update needed — Content Collections picks it up automatically

### Component Organization

```
src/
├── components/
│   ├── blog/           # Blog-specific components (BlogClientWrapper island, BlogPostCard, BlogList)
│   ├── common/         # Shared components (ContactItem)
│   ├── contact/        # Contact page components (ContactForm, ContactInfo)
│   ├── footer/         # Footer sub-components (FooterBrand, FooterLinks)
│   ├── home/           # Home page sections (HeroSection, IntroSection, ServicesSection, etc.)
│   ├── islands/        # React islands requiring client-side hydration (Navbar, BlogClientWrapper)
│   ├── layout/         # Layout sub-components
│   ├── navigation/     # Navigation sub-components (DesktopNav, MobileNav)
│   └── pricing/        # Pricing components (PriceCard, PaymentInfo)
├── layouts/
│   ├── Layout.astro        # Base layout: SEO head, JSON-LD, nav, footer
│   └── ServiceLayout.astro # Shared layout for service pages
└── pages/              # Astro file-based routes (see Routing section above)
```

**Islands rule:** Only components in `src/components/islands/` (and interactive React components) receive `client:*` directives. Everything else is server-rendered `.astro`.

### Custom Hooks

React hooks live in `src/hooks/` and are used within React islands only:

- `useContactForm` - Contact form state and EmailJS submission
- `useScrollToSection` - Smooth scroll to page sections
- `useMotionVariants` - Framer Motion animation variants (respects prefers-reduced-motion)
- `useClipboard` - Copy to clipboard functionality

### State Management

**No global state library** (Redux, Zustand, etc.) - using React hooks only:
- Component state: `useState`
- Reusable logic: Custom hooks
- Data flow: Props drilling (sufficient for current complexity)

### Configuration

- **Global config:** `src/config/global.config.ts` (site metadata, EmailJS keys)
- **Footer links:** `src/config/footer.config.ts` (footer navigation structure)

## Code Conventions

### Language
- **Code & Types:** English
- **User-facing text:** French
- **Commit messages:** French (present tense: "Ajoute" not "Ajouté")
- **Comments:** English for technical, French acceptable for business logic

### File Naming
- **Components:** PascalCase (`BlogPostCard.tsx`)
- **Utilities:** camelCase (`blogApi.ts`)
- **Hooks:** camelCase with "use" prefix (`useBlogPost.ts`)
- **Types:** camelCase (`blog.ts`, `navigation.ts`)

### Component Structure
```typescript
// 1. Imports (React → third-party → local)
import { useState } from "react";
import { motion } from "framer-motion";
import { BlogPost } from "../types/blog";

// 2. Type definitions
interface Props {
  title: string;
}

// 3. Component (function declaration, not arrow)
export default function Component({ title }: Props) {
  // 4. Hooks
  const [state, setState] = useState();

  // 5. Handlers (prefix with "handle")
  const handleClick = () => {};

  // 6. Effects
  useEffect(() => {}, []);

  // 7. Return JSX
  return <div />;
}
```

### TypeScript
- **Strict mode enabled** - no `any` types
- **Interfaces** for object shapes
- **Types** for unions/primitives
- Optional chaining: `user?.profile?.name`
- Nullish coalescing: `title ?? 'Default'`

### Styling (Tailwind CSS)
- **Utility-first only** - no custom CSS except in `src/index.css` for base styles
- **No inline styles** except for dynamic values
- **Organize classes** by category: layout → spacing → typography → colors → responsive

```tsx
<div className="
  flex flex-col           // Layout
  p-4 gap-2              // Spacing
  text-lg font-semibold  // Typography
  bg-white text-gray-900 // Colors
  md:flex-row md:p-6     // Responsive
" />
```

### Accessibility Requirements
- **Semantic HTML:** Use `<nav>`, `<main>`, `<article>`, `<section>` (not `<div>` soup)
- **ARIA attributes:** Label icon buttons, use role/aria-describedby appropriately
- **Keyboard navigation:** Logical tab order, visible focus indicators
- **Alt text:** Descriptive for meaningful images, empty `alt=""` for decorative
- **Test before PR:** Run `npm run audit:a11y` to verify compliance

## Performance Optimizations

### Islands Architecture

Astro renders everything as static HTML by default. React is only loaded for interactive islands:

- `client:load` — hydrate immediately (Navbar, contact form)
- `client:idle` — hydrate when browser is idle
- `client:visible` — hydrate when element enters viewport

### Build Configuration (astro.config.mjs)

- **Static output:** `output: 'static'` — full SSG, no server runtime needed
- **Netlify adapter:** `@astrojs/netlify` — generates `_redirects` and edge functions
- **Sitemap:** `@astrojs/sitemap` — auto-generated at build time
- **Tailwind:** `@astrojs/tailwind` with `applyBaseStyles: false` (managed in `src/index.css`)

### Images
- Use responsive images with multiple sizes
- Prefer modern formats (WebP, AVIF)
- Lazy load below-fold images

## Contact Form

Uses **EmailJS** for client-side email sending (no backend):
- Configuration in `src/config/global.config.ts`
- Form logic in `src/hooks/useContactForm.ts`
- Email template configured in EmailJS dashboard
- Toast notifications for user feedback (react-hot-toast)
